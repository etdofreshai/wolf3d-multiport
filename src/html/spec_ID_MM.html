ID_MM.C — Memory Manager Engine

Purpose: Custom memory manager for the 640KB DOS real-mode environment. Tracks allocations as a linked list of blocks in paragraph-granularity segments. Supports purge levels (allowing cached data to be freed under pressure), locking (preventing relocation), and sorting (defragmentation). Also manages EMS and XMS extended memory.

Author: John Carmack

Key Data Structures

typedef struct mmblockstruct {
    unsigned    start, length;      // segment start and length in paragraphs
    unsigned    attributes;         // LOCKBIT | purge level
    memptr      *useptr;            // pointer to the segment pointer (for relocation)
    struct mmblockstruct far *next;
} mmblocktype;

Porting Notes

C SDL3
The entire segment-based memory manager is replaced with standard malloc/free. The memptr type becomes a regular pointer. Purge levels are tracked but sorting/defragmentation is not needed. The MM_GetPtr/MM_FreePtr wrappers are kept for API compatibility with the rest of the codebase. EMS/XMS code is removed entirely.
C#
Use managed byte[] arrays for all allocations. The GC handles fragmentation. MM_GetPtr becomes new byte[size]. Purge levels can be tracked to know when to null out references for GC collection.
TypeScript
ArrayBuffer / Uint8Array for allocations. JavaScript GC handles lifetime. The block tracking can be greatly simplified or eliminated.
Love2D
Lua tables or FFI-allocated memory. The entire manager becomes a thin wrapper that tracks allocated buffers for the cache system's purge logic.
