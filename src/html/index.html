Wolfenstein 3-D Multiport Source Spec

Faithful port of the original 1992 Wolfenstein 3-D source code to multiple modern languages and platforms

Project Overview

This project takes the original Wolfenstein 3-D source code (released by id Software in 1995) and ports it faithfully to multiple modern languages and runtimes. The goal is not to rewrite or modernize the game design, but to produce functionally identical ports that preserve the original behavior while replacing DOS-specific hardware access (VGA registers, interrupt handlers, EMS/XMS memory, Sound Blaster I/O) with platform-appropriate equivalents.

Architecture

The codebase divides cleanly into two layers:

ID Engine Subsystems (ID_*)

Reusable engine modules originally developed across Keen, Catacomb, and Wolf3D. These handle low-level hardware: video, input, sound, memory management, caching, and page management. Each has a _Startup/_Shutdown lifecycle.

Wolfenstein Game Logic (WL_*)

Game-specific code: main loop, gameplay, rendering, AI, menus, intermission screens, and text display. These call into the ID engine layer but contain no direct hardware access.

Key Porting Challenges

VGA Planar Graphics

The original uses VGA Mode-X with 4 bitplanes. All drawing code writes to specific planes via I/O ports. Modern ports use a linear 320x200 byte framebuffer and convert to RGBA for display.

Real-Mode Memory Model

DOS far pointers, segment:offset addressing, EMS/XMS extended memory. The ID_MM memory manager tracks allocations in 16-byte paragraphs. Modern ports replace this with standard malloc/free.

Inline Assembly

Heavy use of Borland C inline asm blocks and separate .ASM files for performance-critical routines (raycasting, scaling, sound mixing). These must be converted to portable C or the target language.

Hardware Interrupts

Keyboard via INT 9, timer via INT 8, Sound Blaster DMA, AdLib register programming. SDL3 provides event polling and audio callbacks as replacements.

Fixed-Point Math

All game math uses 16.16 fixed-point arithmetic. This carries over naturally to C and C# but needs care in JavaScript/Lua where all numbers are floating-point.
